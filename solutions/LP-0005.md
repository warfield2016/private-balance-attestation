# Solution: LP-0005 — Private Token Balance Attestation

**Submitted by:** warfield2016

## Summary

A reusable private balance attestation primitive for the Logos
Execution Zone. A holder of a shielded LEZ token account generates a
RISC0 proof that `balance >= N` without revealing the balance, the
account, or the nullifier key. The same proof verifies through two
paths against the same `attestation_verifier::verify_attestation`
function: on-chain via a `#[lez_program]` annotated SPEL program, or
off-chain over Logos Messaging.

Forwarding resistance is solved by embedding an ed25519 verification
key in the account's `data` payload, asserting equality with the
public `presenter_pk` inside the circuit, and requiring the presenter
to sign a fresh verifier-issued challenge.

## Approach

### Single verification function, two transports

`attestation_verifier::verify_attestation` is the only place the rules
of admission live. The on-chain SPEL program does not link this crate
(`attestation_program/Cargo.toml` deliberately excludes it); instead
its dispatcher decodes the journal directly, applies the same field-
equality checks, and verifies the ed25519 signature inline. The off-
chain path calls `verify_attestation` directly because it has no LEZ
runtime to lean on for the receipt-cryptographic check. Both paths
converge on identical `JournalFields` policy. Rationale captured in
`ADR.md` ADR-008.

### Why `attestation_program` does not link `attestation_verifier`

Cargo's feature unification means even `default-features = false` on
our direct dep does not stop upstream `nssa_core` from pulling
`risc0-zkvm` with full features. Full features pull `bonsai-sdk` →
`reqwest` → `rustls` → `ring`, which fails to cross-compile to
`riscv32im-risc0-zkvm-elf` (ring's build script passes `-m64`). The
on-chain dispatch was rewritten to do journal decoding + ed25519 sig
verification inline, matching how `lez-multisig` keeps its on-chain
crate dep-light. ADR-008 + ADR-009.

### Equality-bind for presenter binding (the Known Open Problem)

Considered: signature-of-knowledge with ed25519 verify inside the
guest. Adds ~100k cycles per proof and forces the verifier to know
the challenge at proof-gen time. Rejected; equality-bind keeps the
circuit cheap and lets the verifier issue a fresh challenge per
session. SoK is documented in `docs/presenter-binding.md` as
additive future work — the off-chain path doesn't change.

### `program_owner` as a public input (CIRCUIT_VERSION = 2)

Without binding `program_owner` to a public value, a prover with an
account in token program A could satisfy a gate that expects program
B as long as both shared a root. v2 promotes `program_owner` to the
journal and asserts equality in the circuit. ADR-002.

### Trailing-colon domain separators for context-id

Earlier draft had `"lp-0005:"` as the generic prefix. That let
`context_id_generic("chat", group_pk || epoch)` collide with
`context_id_for_chat(group_pk, epoch)`. The trailing colon
(`"lp-0005:chat:"`, `"lp-0005:generic:"`, etc.) makes prefix
collisions impossible by construction. Pinned by a test in
`attestation_core/src/context.rs`. ADR-004.

### Receipt verify before any journal read

`verify_attestation` runs `receipt.verify(image_id)` first, then
reads journal fields. An earlier draft did fast-path policy checks on
the unauthenticated journal — that leaked a policy oracle to
attackers sending forged receipts. ADR-007.

## FURPS Self-Assessment

### Functionality

Six on-chain instructions (`Initialize`, `GateAction`, `RotateAdmin`,
`AddCircuit`, `RevokeCircuit`, `UpdateMinimum`) routed through the
`#[lez_program]` macro to per-instruction handlers in
`attestation_program/src/handlers/`. `CIRCUIT_VERSION = 2`.

### Usability

`make setup && make build idl deploy` from a clean clone produces a
running gate against a localnet. The TypeScript SDK shells out to
`attestation-cli` on Node; a WASM build of the verifier is queued for
v0.2.

### Reliability

32 host-side unit tests across the three workspace crates that
compile without the RISC0 toolchain. Errors are typed in Rust,
surfaced as numeric codes across the FFI boundary.

### Performance

Proof generation: 47s on Linux x86, 5.8s on CUDA, 88s for the
Groth16-wrapped variant. On-chain `GateAction` (Groth16): 94,800 CU.
Receipt size: ~200KB STARK / ~1KB Groth16. See `docs/benchmarks.md`
for methodology + p95s.

### Supportability

Public repo. Three CI jobs pin green on `main`. `make precheck`
mirrors CI exactly so contributors don't push broken commits. Pins
are tagged releases (`nssa_core` v0.2.0-rc3, `spel-framework` v0.3.0,
`risc0-zkvm` =3.0.5). ADRs record every structural decision.
